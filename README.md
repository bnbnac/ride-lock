# ride-lock

배차 매칭 시스템 — 동시성 제어(비관적/낙관적/Redis 분산락) 비교 중심 포트폴리오 프로젝트.

## 3주차: 락 3종 × 스레드 모델 부하테스트

**시나리오**: 승객 100명(k6 VU)이 고정 기사 5명을 놓고 `POST /matchings`를 지속 재시도(think-time 없음, closed workload)하는 핫존 상황. `matching.assign-timeout-seconds=5`로 단축해 기사가 자주 순환하도록 만들었다. 3분씩 6회 실행(`loadtest/run-comparison.sh`).

| 락 전략 | 스레드 모델 | 성공 TPS | 409 실패 수 | 예상외 에러 | 평균 매칭 시간(ms) |
|---|---|---|---|---|---|
| 비관적 | 플랫폼 | 2.08 | 1,768,942 | 0 | 26.60 |
| 비관적 | Virtual | 2.08 | 1,689,927 | 0 | 27.51 |
| 낙관적 | 플랫폼 | 2.08 | 1,753,027 | 0 | 28.83 |
| 낙관적 | Virtual | 2.08 | 1,704,102 | 0 | 28.36 |
| Redis | 플랫폼 | 2.14 | 1,737,670 | 0 | 32.02 |
| Redis | Virtual | 2.08 | 1,655,136 | 0 | 32.18 |

**해석 시 주의**: closed workload 모델이라(VU 100명 고정, 응답 즉시 재요청) TPS가 낮게 나온 조합은 "요청이 적어서"가 아니라 "서버가 그만큼만 처리했다"는 뜻이다. 409는 락 경합에 의한 정상 실패이고, 예상외 에러(5xx)와는 분리해서 봐야 한다.

### TPS가 6개 다 비슷한 이유

성공 TPS는 락 전략과 스레드 모델을 가리지 않고 6개 조합 중 5개가 2.08로 사실상 동일하다(Redis+플랫폼만 2.14로 소폭 높음). 기사가 5명뿐이고 `assign-timeout-seconds=5`로 배차가 반복 순환하는 구조상, 성공 처리량은 락 메커니즘의 처리 속도가 아니라 "기사가 얼마나 자주 비는가"에 의해 상한이 걸린다. 세 락 모두 이 상한보다 충분히 빠르기 때문에, TPS로는 락 간 차이를 볼 수 없는 워크로드다.

### 승객이 실제로 체감하는가: 매칭 시간

평균 매칭 시간은 비관적(26.6~27.5ms) < 낙관적(28.3~28.8ms) < Redis(32.0~32.2ms) 순으로 일관된 격차를 보인다. Redis가 가장 느린 건 락 획득/해제에 드는 네트워크 왕복 비용이 그대로 지연으로 반영된 결과로 설명된다. 다만 이 격차(최대 5~6ms)는 인간이 지연을 지각하는 임계값(대략 100ms)에 한참 못 미친다 — 즉 세 전략 중 어느 것을 쓰든 승객은 매칭 속도 차이를 체감하지 못한다. UX 관점에서 "매칭이 얼마나 빠른가" 축은 사실상 무차별하다는 게 정직한 결론이다.

### 대기형 락이 fail-fast처럼 동작하는 이유

원래 예상은 비관적/Redis(경쟁자를 대기시켰다가 결과를 알려주는 대기형)와 낙관적(대기 없이 즉시 실패, CAS로 재확인)이 체감 지연에서 뚜렷이 갈리는 것이었다. 하지만 `MatchingService`가 트랜잭션 범위를 후보 1명당으로 좁혀놔서(락 경합 측정이 왜곡되지 않도록 하는 설계 의도) `FOR UPDATE`/분산락의 보유 시간 자체가 매우 짧다. 그 결과 대기형 락도 체감상 fail-fast에 가깝게 동작해서, 낙관적 락과의 UX 차이가 숫자로 거의 드러나지 않는다 — 트랜잭션 스코프 설계가 애초에 이 차이를 지운 셈이다.

### 실패 경험: 재시도는 누가, 얼마나 겪는가

`MatchingService.match()`는 후보를 최대 20명(`candidate-limit`)까지 순회하며 서버 내부적으로 재시도하는 구조를 갖고 있다. 하지만 핫존 시나리오는 기사 5명 고정 + 반경 내 IDLE 상태만 후보로 조회하기 때문에, 경합이 심한 시점엔 요청 시점에 후보가 이미 0명인 경우가 대부분이라 이 내부 재시도가 실질적으로 작동할 여지가 거의 없다. 결과적으로 재시도는 거의 전부 클라이언트(k6, think-time 없음) 쪽에서 발생하고, 성공 대비 409 비율은 6개 조합 모두 1:4,500~4,700 수준으로 거의 동일하다(예: 비관적-플랫폼 374성공:1,768,942건, Redis-플랫폼 385성공:1,737,670건). 즉 "얼마나 자주 실패를 겪는가"는 락 전략이 아니라 핫존의 기사 부족 자체가 결정한다. 스레드 모델(플랫폼 vs Virtual) 축에서 플랫폼 쪽 409 수가 항상 소폭 많게 나오지만(3~5%), 조합당 단일 실행이라 이 방향성이 재현 가능한 패턴인지 노이즈인지는 이 데이터만으로 판단할 수 없다.

### 결론

이 워크로드 기준으로 "승객이 체감하는 매칭 경험"은 세 락 전략 간 사실상 무차별하다 — 성공까지 걸리는 시간 차이는 인지 불가능한 수준이고, 실패를 겪는 빈도는 락이 아니라 핫존 구조(기사 5명)가 결정한다. 그러므로 이 프로젝트에서 락 전략을 고르는 근거는 UX보다는 운영 리스크(예: Redis 장애 시 매칭 전체가 멈추는가 vs DB만 살아있으면 동작하는가) 같은 축에서 찾는 게 더 타당하다.

### 측정 한계

- 조합당 단일 3분 실행(n=1)이라 평균만 있고 표준편차/p95가 없다. 26~32ms 격차나 409 수의 플랫폼/Virtual 차이가 실제 패턴인지 단일 실행의 변동인지 이 데이터만으로는 구분할 수 없다.
- `match_success_duration`(k6 Trend)은 200 응답에만 기록된다. 전체 요청의 99.9%+를 차지하는 409 경로의 지연은 측정되지 않아, "실패했을 때 얼마나 오래 기다렸는가"는 이번 데이터에 없다.
