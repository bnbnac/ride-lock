# ride-lock

배차 매칭 시스템 — 동시성 제어(비관적/낙관적/Redis 분산락) 비교 중심 포트폴리오 프로젝트.

## 3주차: 락 3종 × 스레드 모델 부하테스트

**시나리오**: 승객 100명(k6 VU)이 고정 기사 5명을 놓고 `POST /matchings`를 지속 재시도(think-time 없음, closed workload)하는 핫존 상황. `matching.assign-timeout-seconds=5`로 단축해 기사가 자주 순환하도록 만들었다. 3분씩 6회 실행(`loadtest/run-comparison.sh`).

| 락 전략 | 스레드 모델 | 성공 TPS | 409 실패 수 | 예상외 에러 | 평균 매칭 시간(ms) |
|---|---|---|---|---|---|
| 비관적 | 플랫폼 | 2.08 | 1,768,942 | 0 | 26.60 |
| 비관적 | Virtual | 2.08 | 1,689,927 | 0 | 27.51 |
| 낙관적 | 플랫폼 | 2.08 | 1,753,027 | 0 | 28.83 |
| 낙관적 | Virtual | 2.08 | 1,704,102 | 0 | 28.36 |
| Redis | 플랫폼 | 2.14 | 1,737,670 | 0 | 32.02 |
| Redis | Virtual | 2.08 | 1,655,136 | 0 | 32.18 |

**해석 시 주의**: closed workload 모델이라(VU 100명 고정, 응답 즉시 재요청) TPS가 낮게 나온 조합은 "요청이 적어서"가 아니라 "서버가 그만큼만 처리했다"는 뜻이다. 409는 락 경합에 의한 정상 실패이고, 예상외 에러(5xx)와는 분리해서 봐야 한다.

### 이 테스트가 답하는 질문

이 부하테스트는 실제 승객이 체감하는 매칭 경험을 재현하지 않는다. VU 100명이 think-time 없이 즉시 재요청하는 closed workload, 좌표 고정, 기사 5명 고정, 운영값보다 훨씬 짧은 `assign-timeout-seconds=5`를 조합한 인위적 최악의 경합 상황이다. 3분간 409가 170만 건대로 쌓이는 게 그 증거다 — 실제 승객은 이렇게 초당 수천 번씩 재시도하지 않는다. 따라서 이 표의 절대적인 TPS/지연 수치를 프로덕션 용량 산정이나 UX 벤치마크 근거로 쓰면 안 된다.

이 테스트가 실제로 검증하는 건 **지속적인 최악의 동시 경합 상황에서 세 락 전략이 정확성을 지키며 안정적으로 버티는가**다. 그 질문에는 유효하게 답한다: 락 3종 × 스레드 모델 2종, 6개 조합 전부 예상외 에러 0건, 이중 배정 없이(성공 건수만큼만 Trip 생성) 3분간 지속 경합을 견뎠다.

### 관찰된 경향 (참고용 — 일반화 금지)

전략별 순위는 관찰된다 — 평균 매칭 시간 비관적(26.6~27.5ms) < 낙관적(28.3~28.8ms) < Redis(32.0~32.2ms), TPS는 5/6 조합이 2.08로 사실상 동일. 하지만 이 순위를 프로덕션 성능 주장의 근거로 쓸 수 없는 이유가 있다.

- **n=1**: 조합당 단일 3분 실행이라 이 순위가 재현 가능한 패턴인지 단일 실행의 변동인지 통계적으로 구분할 수 없다.
- **성공 경로만 측정**: `match_success_duration`(k6 Trend)은 200 응답에만 기록돼, 전체 요청 99.9%+를 차지하는 409 경로의 지연은 이 데이터에 없다.
- **인위적 조건 종속**: 좌표 고정 + 기사 5명 고정 + 5초 타임아웃이라는 특정 조합에서 나온 순위라, 다른 기사 밀도나 타임아웃 값에서는 순서가 뒤집힐 수도 있다. 성공 대비 409 비율이 6개 조합 모두 1:4,500~4,700로 거의 동일한 것도, 락 전략이 아니라 "기사 5명"이라는 핫존 설정 자체가 재시도량을 결정한다는 뜻이다.

### 결론

이 6회 실행이 뒷받침하는 주장은 하나뿐이다 — 세 락 전략(+2가지 스레드 모델) 모두 인위적인 지속 최악 경합 상황에서 정확성이 깨지거나 예상외 에러 없이 버틴다. 처리량/지연의 절대 수치나 재시도 경험을 실사용자 UX나 프로덕션 성능 근거로 쓰려면, think-time을 넣고 기사:승객 비율을 현실적으로 맞추고 반복 측정을 하는 별도 테스트가 필요하다.
