# ride-lock

배차 매칭 시스템 — 동시성 제어(비관적/낙관적/Redis 분산락) 비교 중심 포트폴리오 프로젝트.

## 3주차: 락 3종 × 스레드 모델 부하테스트

**시나리오**: 승객 100명(k6 VU)이 고정 기사 5명을 놓고 `POST /matchings`를 지속 재시도(think-time 없음, closed workload)하는 핫존 상황. `matching.assign-timeout-seconds=5`로 단축해 기사가 자주 순환하도록 만들었다. 3분씩 6회 실행(`loadtest/run-comparison.sh`). 각 실행 직후 같은 기사의 두 Trip이 활성 구간에서 겹치는지(이중 배정) SQL로 직접 검증한다.

| 락 전략 | 스레드 모델 | 성공 TPS | 409 실패 수 | 예상외 에러 | 평균 매칭 시간(ms) | 이중 배정 |
|---|---|---|---|---|---|---|
| 비관적 | 플랫폼 | 2.08 | 1,870,003 | 0 | 25.35 | 0건 |
| 비관적 | Virtual | 2.08 | 1,793,254 | 0 | 26.10 | 0건 |
| 낙관적 | 플랫폼 | 2.08 | 1,862,523 | 0 | 26.86 | 0건 |
| 낙관적 | Virtual | 2.08 | 1,793,448 | 0 | 25.82 | 0건 |
| Redis | 플랫폼 | 2.14 | 1,867,089 | 0 | 29.48 | 0건 |
| Redis | Virtual | 2.13 | 1,666,986 | 0 | 29.10 | 0건 |

**해석 시 주의**: closed workload 모델이라(VU 100명 고정, 응답 즉시 재요청) TPS가 낮게 나온 조합은 "요청이 적어서"가 아니라 "서버가 그만큼만 처리했다"는 뜻이다. 409는 락 경합에 의한 정상 실패이고, 예상외 에러(5xx)와는 분리해서 봐야 한다.

### 이 테스트가 답하는 질문

이 부하테스트는 실제 승객이 체감하는 매칭 경험을 재현하지 않는다. VU 100명이 think-time 없이 즉시 재요청하는 closed workload, 좌표 고정, 기사 5명 고정, 운영값보다 훨씬 짧은 `assign-timeout-seconds=5`를 조합한 인위적 최악의 경합 상황이다. 3분간 409가 170만 건대로 쌓이는 게 그 증거다 — 실제 승객은 이렇게 초당 수천 번씩 재시도하지 않는다. 따라서 이 표의 절대적인 TPS/지연 수치를 프로덕션 용량 산정이나 UX 벤치마크 근거로 쓰면 안 된다.

이 테스트가 실제로 검증하는 건 **지속적인 최악의 동시 경합 상황에서 세 락 전략이 정확성을 지키며 안정적으로 버티는가**다. 그 질문에는 유효하게 답한다: 락 3종 × 스레드 모델 2종, 6개 조합 전부 예상외 에러 0건. 이중 배정 여부도 더 이상 추정이 아니다 — `run-comparison.sh`가 실행 직후 `trip` 테이블에서 같은 driver_id의 두 Trip이 활성 구간([assigned_at, 만료/현재 시각))에서 겹치는 쌍을 직접 SQL로 세는데, 6개 조합 전부 0건이었다. Unit 레벨 동시성 테스트(`PessimisticLockConcurrencyTest` 등, 스레드 50개·기사 1명)가 증명한 불변조건이 실제 k6 부하(스레드 100개 동시 요청·3분 지속) 하에서도 깨지지 않았다는 걸 DB 상태로 직접 확인한 것이다.

### 관찰된 경향 (참고용 — 일반화 금지)

이 표는 원래 실행을 한 번 더 반복한 결과다. 재실행에서 순위가 어떻게 흔들렸는지가 그 자체로 "일반화 금지"의 증거다.

- **평균 매칭 시간 순위가 스레드 모델에 따라 뒤집혔다**: 플랫폼 스레드에서는 비관적(25.35) < 낙관적(26.86)이 유지됐지만, Virtual 스레드에서는 낙관적(25.82)이 비관적(26.10)보다 오히려 빨랐다. 원래 실행에서는 두 조합 모두 "비관적 < 낙관적"이 일관됐는데, 재실행에서 한쪽이 뒤집힌 것 자체가 이 격차(1ms 안팎)가 락 오버헤드가 아니라 실행별 변동(노이즈)에 가깝다는 뜻이다.
- **Redis만 두 번의 실행에서 일관되게 가장 느렸다**(원래 32.0~32.2ms, 재실행 29.1~29.5ms) — 절대값은 실행마다 달라도 "Redis가 가장 높다"는 순위 자체는 n=2에서 재현됐다. 다만 반복 2회로는 여전히 통계적 신뢰구간을 말할 수 없다.
- **성공 경로만 측정**: `match_success_duration`(k6 Trend)은 200 응답에만 기록돼, 전체 요청 99.9%+를 차지하는 409 경로의 지연은 이 데이터에 없다.
- **인위적 조건 종속**: 좌표 고정 + 기사 5명 고정 + 5초 타임아웃이라는 특정 조합에서 나온 순위라, 다른 기사 밀도나 타임아웃 값에서는 또 달라질 수 있다. 성공 대비 409 비율은 재실행에서도 6개 조합 모두 1:4,300~5,000대로 거의 동일했다 — 락 전략이 아니라 "기사 5명"이라는 핫존 설정 자체가 재시도량을 결정한다는 이전 결론과 일치한다.

### 결론

이 실행들이 뒷받침하는 주장은 두 가지다 — (1) 세 락 전략(+2가지 스레드 모델) 모두 인위적인 지속 최악 경합 상황에서 정확성이 깨지거나 예상외 에러 없이 버틴다, (2) 같은 기사에 대한 이중 배정이 6개 조합 모두 0건이었다(DB 레벨로 직접 검증). 반면 전략 간 지연 순위(비관적/낙관적 사이)는 재실행에서 흔들렸으므로 신뢰할 수 없고, Redis가 가장 느리다는 경향만 약하게(n=2) 재현됐다. 처리량/지연의 절대 수치나 재시도 경험을 실사용자 UX나 프로덕션 성능 근거로 쓰려면, think-time을 넣고 기사:승객 비율을 현실적으로 맞추고 반복 측정을 하는 별도 테스트가 필요하다.

## 3주차 후속: 느린 I/O를 락 안/밖 어디에 두느냐

**동기**: 위 비교에서 세 락 전략 간 지연 차이가 거의 안 보인 이유는, 임계구역이 DB row 하나 읽고 쓰는 것뿐이라 트리비얼하게 짧기 때문이다(`DriverLockStrategy` 인터페이스 주석: "실패 후보의 락 보유 시간은 짧게 유지된다"). 실제 배차 서비스라면 배정 확정 전에 라우팅/ETA 계산처럼 시간이 걸리는 단계가 있을 법한데, 그 단계를 락 안에 두느냐 밖에 두느냐는 실무에서 실제로 중요한 설계 판단이다. 이를 흉내낸 지연(`RoutingClient`/`SimulatedRoutingClient` — 실제 라우팅 엔진 연동은 §6대로 스코프 밖, 인터페이스만 준비해 나중에 교체 가능하게 함)을 추가해 재현했다.

**시나리오**: 기존 핫존 시드(기사 5명) 그대로, `matching.routing-delay-ms=200`(라우팅 API 왕복시간 흉내)을 추가하고 `matching.routing-delay-inside-lock`으로 이 지연을 락 획득 전(before-lock, 기본값)/후(inside-lock) 어디에 둘지 전환한다. 비관적·Redis는 보유할 락이 있어 두 배치를 다 측정했고, 낙관적은 CAS 한 번뿐이라 배치 축이 성립하지 않아 단일 실행만 했다(대신 버전을 읽은 직후~CAS 시도 직전이라는 유일한 자리에 고정 — `OptimisticLockStrategy` 참고). 2분씩 5회 실행(`loadtest/run-routing-delay-comparison.sh` + 낙관적 단일 실행). 비관적 조합은 실행 중 `pg_stat_activity`를 1초 간격으로 캡처해 실제 락 대기 세션을 직접 확인했다.

| 조합 | 성공 TPS | 409 수 | 평균 매칭 시간(성공만, ms) | 전체 요청 평균 지연(ms) | 전체 요청 최대 지연 | 이중 배정 |
|---|---|---|---|---|---|---|
| 비관적 before-lock | 1.82 | 761,742 | 308.8 | 15.67 | 2.69s | 0건 |
| 비관적 inside-lock | 1.84 | 429,724 | 402.4 | 27.84 | **4.45s** | 0건 |
| Redis before-lock | 1.85 | 631,926 | 374.4 | 18.91 | 2.67s | 0건 |
| Redis inside-lock | 1.93 | 1,008,213 | 246.1 | 11.8 | **802ms** | 0건 |
| 낙관적(배치 없음) | 1.82 | 783,565 | 301.0 | 15.23 | 2.72s | 0건 |

### 비관적 락: 예상대로 확실한 안티패턴

inside-lock이 되자 같은 2분 안에 처리된 전체 요청량이 44% 줄었다(76만→43만) — 실패한 요청이 즉시 튕기지 못하고 FOR UPDATE 뒤에서 대기하느라 시도 자체가 느려진 것이다. 평균 매칭 시간은 309→402ms로, 최대 지연은 2.69초→4.45초로 늘었다. `pg_stat_activity` 캡처가 이걸 직접 증명한다 — `wait_event_type=Lock`으로 잡힌 세션이 before-lock에서 27건이던 게 inside-lock에서 450건(16배)으로 뛰었다. "락을 쥔 채 느린 I/O를 하지 마라"는 교과서적 경고가 실제 대기 세션 증거로 재현됐다.

이 캡처는 또 하나를 확인해줬다 — before-lock 캡처에도 `idle in transaction / Client / ClientRead` 상태가 실제로 찍힌다. `tryAssign()` 전체가 `@Transactional`로 감싸여 있어서, before-lock이어도 지연 동안 Postgres 커넥션은 이미 열려 있다는 뜻이다. 즉 "before-lock은 락도 커넥션도 안 쓴다"는 애초 설명은 부정확했고, 정확히는 "row 락만 안 쥔다"였다.

### Redis: 같은 안티패턴인데 다른 방식으로 대가를 치른다

`RedissonLockStrategy`는 `tryLock(0, SECONDS)`로 대기 없이 즉시 실패하는 방식이라(비관적의 FOR UPDATE와 다름), inside-lock이어도 경쟁자가 "기다리다 느려지는" 일이 없다. 오히려 성공한 요청의 평균 매칭 시간은 374→246ms로 **좋아졌다** — 실패할 후보를 탐색하는 데 지연을 낭비하지 않고, 락을 실제로 확보한 뒤에만 지연을 물기 때문이다. 대신 락이 잠겨 있는 창이 넓어진 만큼 그 사이 도착한 요청들이 더 많이 튕겨서 409 수가 63만→101만으로 늘었다. 같은 "락을 오래 쥔다"인데 비관적은 지연 증가로, Redis는 실패량 증가로 대가를 치른다.

### 낙관적: 배치가 없는데도 비관적/Redis의 before-lock과 거의 같은 성적

낙관적이 "락이 없어 위험하다"며 크게 무너질 거라 예상했는데, 실제로는 비관적 before-lock·Redis before-lock과 거의 같은 그룹으로 묶였다(평균 매칭 300ms대, 최대 지연 2.7초대). 이유는 구조적이다 — 낙관적의 흐름(버전 읽기 → 200ms 지연 → CAS 시도)과 비관적 before-lock의 흐름(200ms 지연 → FOR UPDATE 시도)이 "성공/실패 여부와 무관하게 후보 1명당 지연을 무조건 낸 다음 빠른 DB 판정을 받는다"는 점에서 사실상 같기 때문이다. 낙관적의 위험(읽기~쓰기 창에서 값이 바뀔 수 있음)은 이 워크로드에서 두드러지는 재시도 폭증으로 이어지지 않았다.

### 결론

세 락 "종류"가 아니라 두 가지 축이 결과를 갈랐다 — (1) 지연이 승패 판정 **전**에 끝나는가 **후**에 끝나는가, (2) 후라면 그 판정 방식이 패자를 대기시키는가(블로킹) 즉시 튕기는가(논블로킹). 비관적 inside-lock만 "판정 후 + 블로킹"을 동시에 만족해 패자가 남의 지연 시간까지 떠안는 구조라 홀로 최악(최대 지연 4.45초)이었고, Redis inside-lock은 "판정 후"까지는 같지만 논블로킹이라 패자가 지연을 아예 안 물어 오히려 최선(802ms)이었다. 나머지 셋(비관적 before-lock·Redis before-lock·낙관적)은 승패가 확정되기 전에 지연을 이미 다 써버린다는 공통점으로 묶였다 — 낙관적은 "쥐는 자원" 자체가 없어 정확히는 CAS로 결과가 확정되기 전이라고 표현해야 맞다. 이중 배정은 5개 조합 전부 0건으로, 이 극단적 조건에서도 정확성은 깨지지 않았다.
