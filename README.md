# ride-lock

배차 매칭 시스템 — 동시성 제어(비관적/낙관적/Redis 분산락) 비교 중심 포트폴리오 프로젝트.

## 3주차: 락 3종 × 스레드 모델 부하테스트

**시나리오**: 승객 100명(k6 VU)이 고정 기사 5명을 놓고 `POST /matchings`를 지속 재시도(think-time 없음, closed workload)하는 핫존 상황. `matching.assign-timeout-seconds=5`로 단축해 기사가 자주 순환하도록 만들었다. 3분씩 6회 실행(`loadtest/run-comparison.sh`).

| 락 전략 | 스레드 모델 | 성공 TPS | 409 실패 수 | 예상외 에러 | 평균 매칭 시간(ms) |
|---|---|---|---|---|---|
| 비관적 | 플랫폼 | 2.08 | 1,768,942 | 0 | 26.60 |
| 비관적 | Virtual | 2.08 | 1,689,927 | 0 | 27.51 |
| 낙관적 | 플랫폼 | 2.08 | 1,753,027 | 0 | 28.83 |
| 낙관적 | Virtual | 2.08 | 1,704,102 | 0 | 28.36 |
| Redis | 플랫폼 | 2.14 | 1,737,670 | 0 | 32.02 |
| Redis | Virtual | 2.08 | 1,655,136 | 0 | 32.18 |

**해석 시 주의**: closed workload 모델이라(VU 100명 고정, 응답 즉시 재요청) TPS가 낮게 나온 조합은 "요청이 적어서"가 아니라 "서버가 그만큼만 처리했다"는 뜻이다. 409는 락 경합에 의한 정상 실패이고, 예상외 에러(5xx)와는 분리해서 봐야 한다.

성공 TPS는 락 전략과 스레드 모델을 가리지 않고 6개 조합 중 5개가 2.08로 사실상 동일하다(Redis+플랫폼만 2.14로 소폭 높음). 기사가 5명뿐이고 `assign-timeout-seconds=5`로 배차가 반복 순환하는 구조상, 성공 처리량은 락 메커니즘의 처리 속도가 아니라 "기사가 얼마나 자주 비는가"에 의해 상한이 걸린 것으로 보인다. 반면 평균 매칭 시간과 409 발생 수에서는 전략별 차이가 뚜렷하다 — 비관적 락이 가장 낮은 평균 매칭 시간(26.6~27.5ms)을, Redis 분산락이 가장 높은 평균 매칭 시간(32.0~32.2ms)을 보이는데, 이는 Redis 락 획득/해제에 드는 네트워크 왕복 비용이 그대로 지연으로 반영된 결과로 해석할 수 있다. 스레드 모델 축에서는 세 전략 모두 공통적으로 플랫폼 스레드 쪽의 409 실패 수가 Virtual 스레드보다 많다(예: 비관적 1,768,942 vs 1,689,927, 낙관적 1,753,027 vs 1,704,102, Redis 1,737,670 vs 1,655,136) — 이는 성공 처리량에는 영향을 주지 않으면서 동일 시간 동안의 재시도(요청) 총량만 다르게 만든 것으로, Virtual 스레드 스케줄링 특성이 요청 처리 주기에 미세한 영향을 준 것으로 추정된다. 종합하면 이 워크로드에서 락 전략 간 실질적 차이는 TPS가 아니라 경합 비용(평균 매칭 시간)에서 가장 뚜렷하게 드러나며, 다음 단계의 병목 진단은 "성공 건수를 늘리는 방향"보다 "동일 성공량 대비 경합 비용을 낮추는 방향"으로 접근하는 것이 타당해 보인다.
